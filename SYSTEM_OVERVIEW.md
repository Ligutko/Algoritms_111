# Система проекту — Лабораторна робота 1

## Реальний кейс: Інтернет-магазин

Уяви, що ти керуєш інтернет-магазином. У тебе є **10 000 товарів** і тисячі покупців. Що відбувається кожну секунду?

---

## Проблема без алгоритмів

```
Покупець шукає "червоний ноутбук"
→ система перебирає всі 10 000 товарів один за одним
→ знаходить за 3 секунди
→ 1000 покупців одночасно = сервер падає 💀
```

Це і є **лінійний пошук O(n)** — наш `linear_search`.

---

## Що ми побудували і навіщо

### Крок 1 — `utils.py`: Генератор тестових даних
> "Перш ніж щось будувати — нам потрібні тестові дані"

```python
# Реальний світ: тестова база товарів
generate_random_array(10000)    → [3421, 8734, 123, ...]  # хаотичний склад
generate_nearly_sorted_array()  → майже відсортований каталог
generate_reversed_array()       → від найдорожчого до найдешевшого
```

**Навіщо:** без тестових даних ти не знаєш чи працює твій код на великих об'ємах.

---

### Крок 2 — `sorts.py`: Сортування товарів
> "Перш ніж шукати — треба відсортувати"

```
Selection Sort:  бере найдешевший → ставить першим → повторює n разів
                 Як людина яка перекладає картки руками — повільно O(n²)

Quick Sort:      ділить на "дешевші за 500грн" і "дорожчі за 500грн"
                 → ділить кожну половину ще раз → і ще раз
                 Як менеджер який делегує — O(n log n) швидко
```

**Реальний результат:**
```
10 000 товарів:
Selection Sort → 1.08 секунди   ← замовлення протухло 😅
Quick Sort     → 0.008 секунди  ← миттєво ✅
```

---

### Крок 3 — `searches.py`: Пошук товару
> "Покупець ввів ціну — знайди товар"

```
Linear Search:        перебирає всі 10 000 → O(n)
Interpolation Search: "ціна 800грн з 1000? Шукай в кінці!"
                      розумно стрибає одразу до потрібної зони → O(log log n)
```

**Реальний результат:**
```
10 000 товарів:
Linear Search        → 13.50 µs
Interpolation Search → 1.30 µs   ← в 10 разів швидше ✅
```

---

### Крок 4 — `logic.py`: Головна задача — Симетрична різниця
> "У нас два склади. Знайди товари які є тільки на одному."

```
Склад A = [телефон, ноутбук, планшет, навушники, мишка]
Склад B = [телефон, ноутбук, планшет, клавіатура, монітор]

Спільні (є в обох):  [телефон, ноутбук, планшет]  ← ігноруємо
Тільки в A:          [навушники, мишка]             ← треба замовити до B
Тільки в B:          [клавіатура, монітор]           ← треба замовити до A

Результат A △ B = [навушники, мишка, клавіатура, монітор]
```

**Як це робиться всередині:**
```
1. Відсортувати склад B               ← Quick Sort
2. Для кожного товару з A: є він в B? ← Interpolation Search
3. Відсортувати склад A               ← Quick Sort
4. Для кожного товару з B: є він в A? ← Interpolation Search
5. Об'єднати унікальні
```

> `logic.py` використовує `sorts.py` + `searches.py` — як конструктор LEGO.

---

### Крок 5 — `benchmark.py`: Замір швидкості
> "Перед запуском в продакшн — виміряй чи витримає навантаження"

```
Як тест-драйв автомобіля:
10 товарів    → обидва сортування миттєві (різниці немає)
100 товарів   → Quick Sort уже в 2.6× швидший
1000 товарів  → Quick Sort у 20× швидший
10000 товарів → Quick Sort у 135× швидший ← ось чому важливо
```
